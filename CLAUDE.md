# CLAUDE.md

이 문서는 Claude(및 협업자)가 이 저장소에서 작업할 때 따라야 할 기준이다.
긴 가이드가 아니라 **결정의 기준선**이다.

## 프로젝트 개요

어린이/부모용 그림 크리에이티브 플랫폼.
**캐릭터·도안을 직접 만들고**, 만든 작품을 갤러리에 자랑하고, 무료 도안을 받고, 좋아요 기반 랭킹을 본다.

## MVP 범위

**포함**
- **캐릭터/도안 빌더**: 동물 자유 배치 모드 + 얼굴 만들기(부품 조립) 모드
- 텍스트 도구로 한 줄 글자 넣기
- 결과 SVG/PNG 다운로드 (인쇄용)
- 내 작품 저장 / 자랑하기(공개 갤러리)
- 무료 도안 다운로드(운영자 업로드만)
- 좋아요 + 좋아요 기반 랭킹

**제외 (MVP에서 만들지 않는다)**
- **사진 업로드·외부 AI 변환** (어린이 사진 미수집 + API 비용 0원 유지)
- 커뮤니티, 댓글, 팔로우, DM, 채팅
- 결제, 구독, 광고
- 사용자 업로드 도안(=모더레이션 부담)
- 정밀 분석/추천 알고리즘

## 기술 스택

- **Next.js (App Router) + TypeScript**
- **Tailwind CSS** (필요할 때만 shadcn/ui 일부)
- **Vercel** 배포
- **Supabase**: Auth + Postgres + Storage
- **캔버스 빌더**: `konva` + `react-konva` (드래그·리사이즈·회전·터치 지원)

기본값을 벗어나는 도구를 추가하기 전, "Next/Supabase 내장으로 안 되나?"를 먼저 묻는다.

## Storage 정책

- **작품 이미지 (`works`)**: **private 버킷 + signed URL**.
  - 비공개 작품은 RLS만으로 차단되어야 하므로, 영구 public URL을 발급하지 않는다.
  - 공개 작품도 동일 경로 — `is_public` 변화에 즉시 반응한다.
  - signed URL의 만료는 짧게(10–60분). 갤러리 카드 렌더 시점에 발급.
- **무료 도안 (`templates`)**: **public 버킷**.
  - 운영자 업로드만 받으며, 모두에게 동일 접근. CDN 캐싱 자유.
- **자산 라이브러리 (`/public/assets/`)**: 정적 자산. 일회성 생성 후 영구 무료.

작품은 사용자가 캔버스에서 만든 SVG 직렬화 결과로 저장한다.

## 개인정보 원칙 (어린이 서비스)

- 정확한 **생년월일·성별 수집 금지**.
- 어린이 프로필은 **연령대 밴드**만 저장한다: `4-6` / `7-9` / `10-12`.
- 실명·얼굴 식별정보·연락처를 어린이에 대해 저장하지 않는다.
- **사진 업로드 기능 자체가 없음** — 어린이 얼굴/원본 이미지가 서버에 닿지 않는다.
- 공개 갤러리는 **부모가 명시적으로 공개 전환한 작품만** 표시. 기본 비공개.

## 개발 원칙

1. **MVP 범위 밖 기능은 만들지 않는다.** 요청이 와도 v1 메모로 남기고 끝낸다.
2. **한 PR = 한 목적.** 리팩터와 기능을 섞지 않는다.
3. **추상화는 두 번째 사용처가 생긴 뒤에.** 한 곳에서만 쓰는 헬퍼·인터페이스·제네릭 금지.
4. **새 의존성은 신중하게.** 추가 시 PR 설명에 이유 한 줄.
5. **시크릿은 리포에 없다.** `.env.local`만, `.env.local.example`에는 키 이름만.
6. **Supabase `service_role` 키는 서버 코드에서만.** 클라이언트 번들 유입 금지.
7. **외부 API는 클라이언트에서 직접 호출 금지.** 반드시 `app/api/*` 경유.
8. **자산은 `/public/assets/` 한 곳에 두고 명명 규칙을 따른다.** (`README.md` 참고)
9. **DB 변경은 `supabase/migrations/`의 SQL로만.** 콘솔 수동 변경 금지.
10. **`main`은 항상 배포 가능.** 직접 푸시 금지, 작업은 feature 브랜치에서.
11. **폴더 깊이 2단 이내**를 기본으로 한다. 새 폴더는 정당화될 때만.
12. **죽은 코드는 즉시 제거.** 주석 처리한 코드를 커밋하지 않는다.

## UI 원칙

1. **모바일 우선.** 부모 스마트폰에서 1분 안에 결과가 나와야 한다.
2. **한 화면 = 한 행동.** 빌더 화면에서는 만들기만, 결과 화면에서는 보기/저장만.
3. **큰 버튼, 짧은 한글.** 어린이가 옆에서 봐도 이해할 수 있는 문구.
4. **기술 에러 노출 금지.** 실패 시 "다시 시도해 주세요"로 통일.
5. **로딩에는 진행 신호.** 5초 이상 걸리는 작업엔 진행 표시.
6. **인쇄 친화.** 도안은 A4 비율, 흑백 라인 기준으로 다운로드/인쇄 가능해야 한다.
7. **Tailwind 유틸 우선.** 커스텀 CSS 파일은 만들지 않는다(불가피한 경우만).
8. **이미지는 `next/image`로.** 원본 그대로 `<img>` 사용 금지. (단 캔버스 자산은 Konva 내부에서 직접 렌더)
9. **접근성 기본기.** 버튼은 `<button>`, 폼은 `<label>` 연결, 색 대비 충분히.

## 테스트 원칙

1. **MVP는 자동 테스트 강제 아님.** 단, 아래는 반드시 테스트한다.
   - 좋아요 토글 라우트 핸들러
   - 공개/비공개 전환 로직
   - 작품 SVG 직렬화/역직렬화 (캔버스 상태 ↔ 저장 가능한 문자열)
2. **수동 점검 체크리스트**(PR 머지 전 매번):
   - 동물 배치 → 다운로드 흐름 1회 수행
   - 얼굴 빌더 → 다운로드 흐름 1회 수행
   - 비공개 작품이 비로그인 사용자에게 보이지 않음
   - 좋아요가 중복으로 들어가지 않음
   - 모바일 폭(375px)에서 레이아웃 깨짐 없음
3. **테스트 도구는 Next 기본 + Vitest 정도까지만.** E2E는 MVP 이후.
4. **버그를 고칠 땐 같은 버그가 재발하지 않을 최소 테스트를 함께 추가.**
5. **테스트를 위해 프로덕션 코드를 비틀지 않는다.** 테스트 가능성 vs 단순함이 충돌하면 단순함이 우선.

## 작업 진행 방식

- 큰 변경 전: **계획 → 승인 → 구현** 순서.
- 작업 끝나면 3줄 보고: **한 것 / 안 한 것 / 다음 결정 필요한 것**.
- 이 문서와 충돌하는 요청을 받으면, 먼저 충돌을 알리고 결정을 묻는다.
